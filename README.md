# Scalable URL Shortener

A URL shortener built to demonstrate system design fundamentals for backend
interviews: caching, rate limiting, click analytics, and collision-free ID
generation — not just a CRUD wrapper.

## Features

- **Shorten & redirect** — `POST /api/shorten` and `GET /{code}` with custom
  vanity codes supported
- **Caching layer** — Redis if available, in-memory LRU fallback otherwise,
  so redirects avoid a DB hit on cache hit (`X-Cache: HIT/MISS` response header)
- **Rate limiting** — token bucket algorithm per IP (burst + sustained rate),
  implemented as ASGI middleware
- **Click analytics** — every redirect is logged (timestamp, referrer,
  hashed IP) and queryable via `GET /api/stats/{code}`
- **Collision-free codes** — short codes are derived from the DB
  auto-increment ID via base62 encoding, not randomly generated, so there's
  no retry-on-collision loop under concurrent writes
- **Load test script** included to generate real throughput/latency numbers

## Architecture

```
Client -> [Rate Limiter Middleware] -> FastAPI
                                          |
                          cache hit? -----+----- cache miss?
                             |                        |
                        return cached URL      query SQLite/Postgres
                             |                        |
                        (async) log click        cache the result
                             |                        |
                        302 redirect  <---------------+
```

In production on AWS this maps to: API Gateway + Lambda (or ECS), DynamoDB
or RDS for storage, ElastiCache (Redis) for the cache layer, and SQS to
decouple click-logging from the redirect path.

## Running it on your system

### Option A: Plain Python (fastest to get started, no Redis)

Requires Python 3.10+.

```bash
cd url-shortener
python3 -m venv venv
source venv/bin/activate        # on Windows: venv\Scripts\activate
pip install -r requirements.txt

uvicorn app.main:app --reload
```

The API is now live at **http://127.0.0.1:8000**.
Interactive docs (auto-generated by FastAPI) at **http://127.0.0.1:8000/docs**.

Without Redis running, the cache layer automatically falls back to an
in-memory LRU cache — everything still works, you just lose the cache
across restarts.

### Option B: Docker Compose (closer to production, includes Redis)

Requires Docker Desktop installed and running.

```bash
cd url-shortener
docker-compose up --build
```

This starts both the app (port 8000) and a real Redis instance, wired
together automatically via `REDIS_URL`.

### Try it out

```bash
# Shorten a URL
curl -X POST http://127.0.0.1:8000/api/shorten \
  -H "Content-Type: application/json" \
  -d '{"url": "https://www.anthropic.com"}'

# Response: {"short_code": "0000001", "short_url": "http://127.0.0.1:8000/0000001", ...}

# Follow the redirect
curl -i http://127.0.0.1:8000/0000001

# Check analytics
curl http://127.0.0.1:8000/api/stats/0000001

# Custom vanity code
curl -X POST http://127.0.0.1:8000/api/shorten \
  -H "Content-Type: application/json" \
  -d '{"url": "https://www.anthropic.com", "custom_code": "claude"}'
```

Or just open **http://127.0.0.1:8000/docs** and use the Swagger UI to try
every endpoint from the browser.

### Run the tests

```bash
pytest tests/ -v
```

### Generate real performance numbers (for your resume/interviews)

With the server running in one terminal, run this in another:

```bash
python load_test.py --requests 500 --concurrency 20
```

This prints throughput (req/sec) and p50/p95/p99 latency — genuine numbers
you measured yourself, not made up ones.

## Talking points for interviews

- **Why base62 IDs instead of random strings?** Avoids collision-retry
  loops under concurrent writes; codes are dense and short.
- **Why a separate Click table instead of a counter column?** Keeps
  analytics writes from locking/contending with the URL table, and makes
  it easy to swap in a time-series store later without touching URL data.
- **Why token bucket over fixed-window rate limiting?** Allows legitimate
  bursts while still capping sustained rate — fixed windows let clients
  double their effective rate at window boundaries.
- **What would you change for real scale?** Move click-logging off the
  request path (SQS + worker), move from SQLite to DynamoDB/Postgres,
  add a CDN/edge cache for redirects, and shard by short_code prefix if a
  single DB becomes a bottleneck.
